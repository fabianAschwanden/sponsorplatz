## Entwicklungsprozess (TDD-Pflicht)

Jedes Feature und jeder Bugfix folgt diesem Dreischritt — **immer in dieser Reihenfolge**:

```
1. SPEC   → Verhalten in specs/ beschreiben
2. TEST   → Test schreiben (red)
3. IMPL   → Implementierung schreiben (green)
```

> **Regel:** Kein produktiver Code ohne vorherigen Test. Kein Test ohne vorherige Spec.

### Schritt-Details

**Spec:** Anforderung in passender Datei unter `specs/` ergänzen. Bei API-Änderungen: Routen-Tabelle in `TECHNISCHE_SPEZIFIKATION.md` aktualisieren.

**Test:** Testfall-ID vergeben (nächste freie nach Konvention SP-xx / OR-xx / AC-xx in `TESTSTRATEGIE.md`). Zuerst sicherstellen, dass der Test fehlschlägt.

**Impl:** Minimale Implementierung, die den Test grün macht. Danach `mvn test` — alle müssen grün sein.

## Clean Code Prinzipien

Code folgt den Clean-Code-Regeln (Robert C. Martin), gleichwertig neben TDD.

### Namen

- Klassen, Methoden, Variablen sagen **was** sie sind oder tun
- Keine Abkürzungen (`sponsoringPaket`, nicht `sp`)
- Booleans als Aussage: `istVerifiziert`, `kannEditieren`, `hatFehler`

### Methoden

- Eine Aufgabe pro Methode
- Maximale Einrückungstiefe: 2–3 Ebenen — sonst private Hilfsmethode
- Guard Clauses statt tiefes `else`-Nesting

```java
// SCHLECHT
public void verarbeite(Anfrage a) {
    if (a != null) {
        if (a.istEingereicht()) { /* tief */ }
    }
}

// GUT
public void verarbeite(Anfrage a) {
    if (a == null || !a.istEingereicht()) return;
    /* flach */
}
```

### Kommentare

- Kein Kommentar für **was** der Code tut — das erklären die Namen
- Kommentare nur für **warum**: versteckte Einschränkung, Workaround, nicht-offensichtliche Invariante

### Konstanten

- Alle Model-Attribute-Keys in `ModelAttributeNames`
- Enum-Werte bleiben Enums, nie als String vergleichen
- Konfiguration als `private static final` Konstanten

### DRY

- Doppelte Logik → private Hilfsmethode oder gemeinsamer Service
- Copy-paste ist Warnsignal

### Fehlerbehandlung

- Services werfen spezifische Exceptions (`IllegalArgumentException`, `IllegalStateException`)
- Controller fangen **keine** Business-Fehler — `GlobalExceptionHandler` übernimmt
- Ausnahmen bewusst und dokumentiert

### Klassen

- Single Responsibility
- Services nach Fachbereich
- Entities (`@Entity`) verlassen den Service-Layer nicht — Controller arbeiten mit DTOs

## Test-Ebenen

| Ebene | Werkzeug | Abdeckung |

| Unit | JUnit 5 + Mockito + AssertJ | Services, Konverter, Validierung |

## Konventionen

### Sprache

- **Code (Klassen, Variablen, Methoden):** Deutsch (`Sponsor`, `Verein`, `kannEditieren`)
- **Logs:** Deutsch
- **Commits:** Deutsch oder Englisch — konsistent pro Person

### Commits

Konvention: `typ(bereich): kurze beschreibung`

- `feat(crm): Sponsor-CRUD mit Validierung`
- `fix(security): Logout setzt Session korrekt invalidiert`
- `docs(readme): Quickstart-Sektion ergänzt`
- `test(integration): SponsoringAnfrage-Workflow Happy-Path`
- `refactor(service): Mail-Versand in eigene Klasse extrahiert`
- `chore(deps): Spring Boot 3.4.1`

### Branches

- `main` — immer deployable
- `feature/<thema>` — kurzlebig (max 5 Tage)
- `fix/<thema>`
- `chore/<thema>`

### Pull Requests

- Mind. ein „Self-Review" (Code lesen mit anderem Hut auf)
- Tests grün, Build grün
- Spec aktualisiert
- Konflikte mit `main` rebased, nicht gemerged

## Häufige Fehler

| Problem | Ursache | Lösung |
|---|---|---|
| Port 8080 belegt | andere App läuft | `lsof -i :8080` → `kill PID` |
| Tests schlagen fehl | veraltete Build-Artefakte | `mvn clean test` |
| H2-Konsole nicht erreichbar | falsches Profil | sicherstellen `dev`-Profil aktiv |
| Flyway-Konflikt nach Schema-Änderung | Hibernate-DDL-Auto hat verändert | DB löschen oder Flyway-Repair |
